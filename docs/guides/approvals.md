# Approvals Guide

Approvals are the main safety boundary in OPENCLAW. This guide explains what is
staged, how review works, and how contributors should extend the system.

## What requires approval?

The following action types are considered mutations:

- `file_create`
- `file_modify`
- `file_delete`
- `folder_create`
- `tool_execute`

Read and analysis actions are logged as executed but do not require approval.

## CLI approval flow

1. The orchestrator calls `runApprovalFlow(tracker)`.
2. Pending mutation actions are grouped by file path or shell command.
3. The user chooses whether to approve all, review one by one, or cancel.
4. File groups can show unified diffs.
5. Each action is marked approved or rejected.
6. The orchestrator calls `applyApprovedFromTracker()`.

## Telegram approval flow

1. Telegram runner finishes an Agent or Plan workflow.
2. `finishOrApprove()` checks pending actions.
3. An in-memory approval session is stored by chat ID.
4. The bot sends inline buttons for diff, accept all, and reject all.
5. Callback handlers apply or clear the session.

## Diff behavior

File diffs are generated by composing before/after text from actions for the
same path and passing the result to `createTwoFilesPatch` from the `diff`
package.

Folder creation and shell execution do not have file diffs. They are displayed
as summary rows.

## Best practices for reviewers

- Read every shell command before approving.
- Inspect diffs for unexpected credential or generated file changes.
- Reject broad changes that are hard to understand.
- Prefer small agent tasks that produce small approval sets.
- Run tests after applying changes.

## Best practices for contributors

- Keep approval summaries clear and specific.
- Add previews for new mutation types.
- Do not make approval optional for mutating tools.
- Preserve owner authorization in Telegram callbacks.
- Add tests for approve, reject, and cancel paths.
